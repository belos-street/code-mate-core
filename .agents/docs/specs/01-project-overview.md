# 01 — 项目概述与技术约束

> **所属 Phase**：全部
> **依赖**：无

---

## 1. 项目目标

将 TypeScript 版本的语法高亮引擎（[ts/lib/](../../../ts/lib/)）复刻为 Go 实现，编译为 WASM（WebAssembly）供前端 JavaScript 环境调用。

## 2. 核心质量指标

- **检测结果一致性**：相同输入代码必须产生相同的 token 流和 HTML 输出（与 TS 版本一致）
- **不要求代码逐行对应**：Go 版本可以有自己的实现风格和优化，只需保证输入/输出行为一致

## 3. 输入/输出规格

| 项目 | 规格 |
|------|------|
| 输入 | 源代码字符串 + 语言标识符（如 `"javascript"`、`"js"`） |
| 输出 | HTML 字符串（带内联样式的语法高亮结果） |
| 中间产物 | TokenStream（二维 Token 数组，按行分组） |

---

## 4. 技术约束

### 4.1 Go 版本

- 使用 Go 1.21+（支持泛型，但本项目不需要泛型，统一用 string）

### 4.2 依赖管理

- 使用 Go Modules
- **零外部依赖**：只使用标准库（`regexp`, `strings`, `sync`, `syscall/js` 等）

### 4.3 正则表达式

- 使用标准库 `regexp`（RE2 语法）
- 所有正则必须预编译（包级变量或 init 函数）
- 不使用 `(?<=...)`, `(?=...)`, `\1` 等不兼容特性

### 4.4 并发安全

- WASM 环境是单线程的，不需要 mutex
- 但代码设计应保留扩展性（如语言注册表可选择性加锁）

### 4.5 性能目标

- 首次解析性能：与 TS 版本持平或更优
- 缓存命中时：接近零开销
- WASM 二进制大小：目标 < 5MB（gzip 后 < 1MB）
- **体积基线**：17 种语言的正则规则 + 主题样式数据量较大，建议 Phase 1-2 完成后（仅 JavaScript + 核心引擎）先编译一次测量基线体积，再评估是否需要对正则做压缩或对非核心语言做延迟加载
